# Contributing to Kinetic-RT

Welcome to Kinetic-RT! We appreciate your interest in contributing. This document provides guidelines for contributing to ensure a smooth, professional, and collaborative development environment.

## Project Philosophy

Kinetic-RT aims to be a modern, high-performance, and professionally maintained open-source systems project. We value clean architecture, maintainability, robust testing, and clear documentation. Our goal is to foster a welcoming community where both new and experienced engineers can collaborate effectively.

## Development Workflow

1. Fork the repository and clone your fork locally.
2. Create a new branch for your feature or bugfix.
3. Write clean, modular, and well-documented code.
4. Add or update tests as necessary.
5. Ensure all tests pass locally.
6. Commit your changes using Conventional Commits.
7. Push your branch and open a Pull Request (PR) against the main repository.

## Branch Naming Conventions

Please use descriptive branch names that reflect the nature of your changes. We recommend the following format:
* `feature/<short-description>` for new features
* `bugfix/<short-description>` for bug fixes
* `docs/<short-description>` for documentation updates
* `perf/<short-description>` for performance improvements
* `test/<short-description>` for test updates

## Commit Message Format

We strictly follow the [Conventional Commits](https://www.conventionalcommits.org/) specification. This leads to more readable messages that are easy to follow when looking through the project history.

**Format:**
```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

**Common Types:**
* `feat`: A new feature
* `fix`: A bug fix
* `docs`: Documentation only changes
* `style`: Changes that do not affect the meaning of the code (white-space, formatting, missing semi-colons, etc)
* `refactor`: A code change that neither fixes a bug nor adds a feature
* `perf`: A code change that improves performance
* `test`: Adding missing tests or correcting existing tests
* `chore`: Changes to the build process or auxiliary tools and libraries

## Pull Request Process

1. Ensure your PR description clearly describes the problem and solution.
2. Fill out the provided Pull Request Template completely.
3. Link any relevant issues.
4. Ensure all CI checks pass.
5. Request a review from code owners or relevant maintainers.

## Coding Standards

### Python Style Expectations
* Follow [PEP 8](https://peps.python.org/pep-0008/) style guidelines.
* Use type hints for all function signatures.
* Ensure code is clean, readable, and properly modularized.
* Use the standard `logging` module for diagnostics, avoiding global `print()` statements in production code.

### C++ Style Expectations
* Write modern C++ (C++17 or newer preferred).
* Follow established conventions for naming, formatting, and memory management within the codebase.
* Avoid raw pointers where possible; prefer smart pointers and robust RAII patterns.
* Do not swallow exceptions. Catch specific exceptions and handle them appropriately, using structured logging or `std::cerr`.
* Keep the core engine strictly model-agnostic.

### Documentation Expectations
* Ensure any new features or API changes are documented.
* Write concise and technically accurate comments. Avoid commemorative meta-comments, debugging tags, or lazy history markers.
* Keep documentation professional, friendly, and easy to maintain without unnecessary corporate or legal fluff.

## Testing Requirements
* All new features and bug fixes must be accompanied by relevant tests.
* Ensure rigorous native C++ testing alongside thorough Python boundary testing (using `pytest`).
* Do not bypass testing requirements.
* Maintain the integrity of existing test suites and avoid breaking legacy tests without proper justification.

## Code Review Process
* All changes require at least one approval from a designated code owner or maintainer before merging.
* Reviewers will evaluate the code for correctness, performance, maintainability, style, and adherence to project philosophy.
* Address all review feedback promptly and update the PR as needed.

## Definition of Done
A contribution is considered "Done" when:
* Code changes are fully implemented without placeholders or fake implementations.
* All relevant unit and integration tests are added/updated and pass.
* Documentation is updated and accurately reflects the implementation.
* The PR has been reviewed and approved by maintainers.
* CI/CD pipelines run successfully.

## AI-Assisted Contribution Policy

We recognize the value of AI tools in modern software development.
* AI-generated code is allowed.
* Contributors are solely responsible for understanding, testing, and verifying the submitted code.
* Placeholder or fake implementations generated by AI are not acceptable.
* Every functional change must be reviewed by humans.
* Documentation must accurately reflect the actual implementation, avoiding AI-generated fluff or inaccuracies.

## Project Scope

Kinetic-RT is an inference runtime.

This repository prioritizes:

- correctness
- maintainability
- performance
- reproducibility

Feature completeness is never prioritized over correctness.

Changes that introduce placeholder implementations, misleading behavior, or undocumented functionality will not be accepted.
